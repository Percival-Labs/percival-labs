# Drizzle snapshot chain repair (2026-07-01)

## What was broken

`drizzle/meta/_journal.json` had 3 entries (`0000`, `0001`, `0020`) and `drizzle/meta/`
only had 2 snapshot files (`0000_snapshot.json`, `0001_snapshot.json`), while `drizzle/`
had 26 hand-authored `*.sql` migrations (`0000`-`0025`). Migrations `0002`-`0025` were
never generated via `drizzle-kit generate`, so their snapshot files were never written and
the journal was never advanced past `0020`.

`drizzle-kit generate` (0.30.6, the version pinned in this package) picks the "previous"
snapshot to diff against by listing `drizzle/meta/*.json`, filtering out `_journal.json`,
and sorting **the filenames** — it does not consult `_journal.json` for this. With only
`0000_snapshot.json` and `0001_snapshot.json` present, it was always diffing current
`src/schema/*.ts` (63 tables) against the `0001` snapshot (22 tables, February), so every
`generate` produced a wall of "is `btc_price_snapshots` renamed from `cosign_proofs`?"
style prompts and hung waiting for interactive input.

## What was NOT broken (and left untouched)

Every existing `drizzle/*.sql` file is byte-for-byte unchanged. No database was connected
to or migrated. Only `drizzle/meta/_journal.json` was rewritten, and one new snapshot file
(`drizzle/meta/0025_snapshot.json`) was added.

## The repair

1. **`_journal.json` rebuilt**: one entry per real `drizzle/*.sql` file, `idx` 0-25 in
   filename order, `tag` = filename, `when` = each file's mtime (already monotonically
   increasing and consistent with the 3 timestamps the old journal did have for `0000`,
   `0001`, `0020`).
2. **New baseline snapshot (`0025_snapshot.json`)**: generated by running
   `drizzle-kit generate` against current `src/schema/*.ts` in a scratch directory with an
   *empty* migrations folder (no prior snapshot, so there is nothing to diff against and no
   interactive prompts are possible), which produces an accurate structural snapshot of the
   full current schema. `id`/`prevId` were then set to chain from the real `0001` snapshot
   (`prevId` = `0001`'s `id`), and the file was placed as `0025_snapshot.json` so it sorts
   after `0001_snapshot.json` and becomes the diff target for the next `generate`.

## A real drift was found, not just a metadata gap: `accounts`, `outcomes`, `royalty_payments`

Before trusting "current schema == migration history," every table `CREATE TABLE`d (net of
any later `DROP TABLE`) across all 26 `*.sql` files was diffed against every table defined
in `src/schema/*.ts`. Two tables matched in both directions except three:

- `accounts` (`src/schema/accounts.ts`)
- `outcomes` (`src/schema/outcomes.ts`)
- `royalty_payments` (`src/schema/skills.ts`)

**None of these three tables has ever had a `CREATE TABLE` statement in any committed
migration.** `accounts` and `outcomes` are only ever referenced via `ALTER TABLE` (e.g.
`0020_vouch_identity_accounts.sql`, `0025_..._and_hardening.sql` add columns to `accounts`;
`0010_outcome_dedup.sql` adds an index to `outcomes`) — both migrations assume the table
already exists. `git log` confirms `outcomes.ts` and `accounts.ts` are not new: `outcomes.ts`
has existed since the original Vouch rebrand, and `accounts.ts` since March 7. This is
long-standing, pre-existing drift, not something introduced by this repair.

Because `accounts` is `ALTER TABLE`'d by two already-committed migrations, **it is very
likely the `accounts` table already exists in any database those migrations were actually
run against** — created some other way (direct SQL, `drizzle-kit push`, etc.) outside the
tracked migration history. The same cannot be said with confidence for `outcomes` or
`royalty_payments`.

The 3 excluded tables (and the 5 enum types used only by them: `account_status`,
`account_plan`, `outcome_role`, `outcome_credit`, `royalty_status`) were deliberately left
**out** of the reconstructed `0025_snapshot.json`, so that running `generate` afterward
would surface this real, pre-existing gap as an honest single migration rather than paper
over it.

### `bun run generate` after the repair

```
$ bun run generate
...
[✓] Your SQL migration file ➜ drizzle/0026_striped_clint_barton.sql
```

`0026_striped_clint_barton.sql` contains exactly: `CREATE TYPE` for the 5 enums above, then
`CREATE TABLE` for `outcomes`, `royalty_payments`, `accounts`, plus 2 FK constraints on
`royalty_payments`. Nothing else — no renames, no touches to any of the other 60 tables.
Running `generate` again immediately afterward reports `No schema changes, nothing to
migrate`, confirming the chain is now internally consistent and idempotent.

**`0026_striped_clint_barton.sql` was generated as part of verifying this repair, but it
has NOT been applied to any database and should not be run against one without a human
decision first.** It does not use `IF NOT EXISTS` guards. If `accounts` already exists in a
real database (likely, per above), running this migration there will fail outright on the
`CREATE TABLE "accounts"` statement. Before running `bun run migrate` anywhere:

1. Check whether `accounts` / `outcomes` / `royalty_payments` already exist in that
   database (`\dt` in `psql`, or `drizzle-kit introspect` against it).
2. If they exist: either mark `0026` as already-applied in that database's migration
   bookkeeping without running it, or hand-edit a **local, not-yet-committed** copy of
   `0026` to add `IF NOT EXISTS` guards and reconcile column-by-column before applying.
3. If they don't exist: review `0026` normally and apply it.

## Known limitation of this repair

Table-level agreement between `src/schema/*.ts` and `drizzle/*.sql` (does each table that
exists in one exist in the other, net of `DROP TABLE`) was verified directly from the SQL
text and is reliable. **Column-level** agreement for the 60 tables that *do* have a tracked
`CREATE TABLE` was not exhaustively verified — that requires either a real Postgres
instance to `introspect` (out of scope: this repair does not touch any database) or a full
SQL-DDL replay of all 26 migrations, which was attempted with a regex-based reconstruction
and abandoned as unreliable (it produced false positives on ordinary multi-line `CREATE
TABLE` bodies). `bun run db:check` (see `README.md`) will catch any *future* drift the
moment it's introduced; it does not retroactively guarantee the pre-existing 60 tables are
column-perfect. Recommend running `drizzle-kit introspect` against a real staging DB at some
point to fully close this out.
