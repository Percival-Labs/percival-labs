## 3. Patent vs. Trade Secret: Trust-Staking-for-API-Access

### 3.1 The Core Innovation

The novel mechanism: **trust staking as a prerequisite for API access**, where AI agents must stake reputation/value to gain access to services, with stakes slashed on misbehavior. This creates economic alignment between API providers and consumers without traditional authentication gatekeeping.

### 3.2 Analysis Matrix

| Factor | Patent | Trade Secret | Winner |
|--------|--------|-------------|--------|
| **Cost** | $2,500-5,000 provisional, $10,000-15,000 non-provisional | $0 (just maintain confidentiality) | Trade Secret |
| **Duration** | 20 years from filing | Indefinite (while secret) | Trade Secret |
| **Disclosure** | Must publish full specification | No disclosure required | Trade Secret |
| **Enforcement** | Can sue infringers, injunctive relief | Requires proving misappropriation | Patent |
| **Open source conflict** | Contradicts open-source philosophy | Compatible if core algo stays proprietary | Trade Secret |
| **C > D test** | Creates temporary monopoly (defection) | Protects while enabling cooperation | Trade Secret |
| **Competitor deterrent** | Prevents others from patenting same thing | Others can independently discover | Patent |
| **AI protocol patentability** | Uncertain — Alice/Mayo framework makes software patents harder | N/A | Trade Secret |
| **Speed** | 2-4 years to grant | Immediate | Trade Secret |

### 3.3 Recommendation: Hybrid Strategy

**Primary**: Trade Secret for core trust-scoring algorithms and staking engine internals.

**Secondary**: Defensive Publication for the protocol-level innovation (trust staking for API access concept).

**DO NOT PATENT** the core concept. Here's why:
1. **Open-source conflict**: Vouch SDK is public on npm. The protocol is described in public docs. You've already published prior art against yourself for some claims
2. **Alice/Mayo problem**: Software methods for economic transactions face the highest bar for patent eligibility under current USPTO guidance
3. **Cost prohibitive**: $10K-15K for a non-provisional you may not be able to enforce
4. **C > D violation**: Patenting a trust protocol that's meant to be a public good undermines the mission

### 3.4 What to Keep as Trade Secret

- Exact trust score calculation algorithms (the weights, decay functions, normalization)
- Staking engine optimization parameters
- Dispute resolution heuristics
- Any ML models trained on trust data
- Internal scoring rubrics and threshold values

**Reasonable measures to maintain trade secret status:**
- Keep scoring internals in private repos (NOT in the public vouch-sdk)
- Add trade secret notices to internal code headers
- Limit access to scoring algorithm source code
- Document the secrecy measures (creates legal standing)

### 3.5 Defensive Publication Strategy

**Purpose**: Prevent competitors from patenting trust-staking-for-API-access and blocking you.

**How to execute (free):**

1. **Publish a technical whitepaper** on arXiv, SSRN, or your own blog describing the trust-staking-for-API-access protocol at a conceptual level
2. **File with IP.com Prior Art Database** — $200 per publication, indexed by USPTO examiners
3. **Leverage existing public artifacts** as prior art:
   - `@percival-labs/vouch-sdk@0.1.0` published Feb 22, 2026 on npm (timestamped, immutable)
   - GitHub commit history in `Percival-Labs/vouch-sdk` and `Percival-Labs/vouch-api` repos
   - Deployed API at `percivalvouch-api-production.up.railway.app`
   - NIP-85 specification references
   - Public tweets about Vouch (Feb 22, 2026 — timestamped on X)

4. **Write a formal defensive disclosure** covering:
   - Trust staking as a mechanism for AI agent API access control
   - Reputation-based access tiers derived from stake history
   - Stake slashing on verified misbehavior as enforcement mechanism
   - Nostr-native identity binding for decentralized trust verification
   - Economic game theory of cooperation incentives in agent-to-agent commerce

**Timeline**: Publish defensive disclosure within 2 weeks (before any outreach to Anthropic/OpenAI).

## 4. Open Source License Strategy

### 4.1 Current State

| Product | Current License | Repo Visibility |
|---------|----------------|-----------------|
| vouch-sdk | Unknown/unlicensed? | Public |
| vouch-api | Unknown/unlicensed? | Public |
| engram-harness | Unknown/unlicensed? | Public |
| Website | N/A | Private |

**CRITICAL WARNING**: If your public repos have no LICENSE file, they default to **all rights reserved**. This means nobody can legally use, modify, or distribute the code — which contradicts your "open by default" philosophy AND creates a chilling effect on adoption. Fix this immediately.

### 4.2 Recommended License Architecture

| Product | License | Rationale |
|---------|---------|-----------|
| **vouch-sdk** | **Apache 2.0** | Maximum adoption. SDK must be as frictionless as possible. Patent grant protects users. |
| **vouch-api** | **AGPL-3.0 + Commercial License** (dual license) | AGPL prevents cloud providers from running your API as a service without contributing back. Commercial license for enterprises who can't comply with AGPL. |
| **engram-harness** | **Apache 2.0** | Framework/CLI adoption requires permissive licensing. |
| **Trust scoring engine** (internal) | **Proprietary / Not published** | Trade secret. Never open-source the scoring internals. |

### 4.3 Why AGPL for the API, Apache for the SDK

The AGPL is specifically designed for the Vouch API use case — it requires anyone who runs a modified version of the API as a network service to release their source code under the same license. This:
- Prevents Amazon/Cloudflare from offering "Managed Vouch" without contributing
- Keeps the protocol open for self-hosters and researchers
- Creates natural commercial license demand from enterprises who refuse AGPL

The SDK should be Apache 2.0 because:
- SDKs need to be embeddable in any project, including proprietary ones
- AGPL/GPL contamination in an SDK would kill adoption
- Apache 2.0 includes an express patent grant (protects your users)

### 4.4 Contributor License Agreement (CLA)

**Required before accepting external contributions.** Without a CLA:
- Contributors retain copyright on their contributions
- You cannot change the license later
- You cannot offer commercial licenses (dual licensing breaks)

**Recommendation**: Use the **Apache-style Individual CLA** (modeled on Google/Apache Foundation templates). Key provisions:
- Contributor grants you a perpetual, worldwide, non-exclusive copyright license
- Contributor grants you a patent license for their contribution
- Contributor represents they have the right to grant these licenses
- Does NOT require copyright assignment (less friction than FSF-style)

**Implementation**: Use [CLA Assistant](https://cla-assistant.io/) — free GitHub integration that requires CLA signature before PR merge.

## 5. Entity Formation

### 5.1 Current Situation

You're operating under **Carroll Carpentry LLC** — a construction/handyman business entity. This creates several problems:

1. **IP ownership ambiguity**: Is the IP owned by you personally or by Carroll Carpentry LLC? If the LLC, that means a carpentry company owns AI software IP — messy for investors, partners, and enforcement
2. **Liability contamination**: Carroll Carpentry's operational liabilities (construction accidents, contract disputes) could theoretically reach PL IP if it's held by the same entity
3. **Investor readiness**: Zero VCs will invest in a carpentry LLC that also does AI trust protocols
4. **Trademark confusion**: Filing trademarks under Carroll Carpentry LLC signals the wrong industry

### 5.2 Recommended Entity Structure

**Phase 1 (NOW — before trademark filing):**

Form **Percival Labs LLC** in your state of residence.
- Cost: $50-300 depending on state
- Timeline: Same day (online filing in most states)
- Purpose: Own all PL intellectual property, hold trademarks, execute CLAs
- Operating agreement: Single-member, you as sole manager

**Phase 2 (When seeking investment or revenue exceeds $50K):**

Convert to or form **Percival Labs, Inc.** as a Delaware C-Corp.
- Cost: $89 (Delaware filing) + $300/yr franchise tax minimum + registered agent ($50-150/yr)
- Use Stripe Atlas ($500) for turnkey incorporation with banking
- IP assignment from LLC to Corp via formal assignment agreement
- This is when you'd also do an 83(b) election on founder shares

**Phase 3 (Post-investment):**

Consider an IP holding company structure where Percival Labs IP LLC holds all IP and licenses it to Percival Labs, Inc. (the operating company). This is the gold standard for IP protection but overkill until you have meaningful revenue or investment.

### 5.3 IP Assignment Chain

The critical step most founders miss: **formally assign all existing IP from yourself to Percival Labs LLC.**

Required document: **Intellectual Property Assignment Agreement** covering:
- All copyrights in Vouch, Engram, Percival Labs codebases
- All trade secrets (scoring algorithms, etc.)
- All trademark rights (common law rights from use)
- All domain names (percival-labs.ai, etc.)
- Assignment of npm package ownership
- Assignment of GitHub organization ownership

This must be a signed, written document. It should be executed the day you form the LLC. Consideration can be membership interest in the LLC.
